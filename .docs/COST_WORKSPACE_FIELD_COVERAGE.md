# Cost Workspace Field Coverage Matrix

Last updated: 2026-05-25

Purpose: make sure Cost Workspace stores enough information to become
PartCatalog Item/Term candidates without hiding costs or losing traceability.
This is the planning source before designing `CostWorkspaceRun`,
`CostWorkspaceLine`, and `CostWorkspaceSnapshot`.

## 1. Storage Layers

| Layer | Meaning | Can overwrite? | Creates revision? |
|---|---|---:|---:|
| Draft / RAM | Current editable workspace state | Yes | No |
| Save Revision | Immutable snapshot of current draft + result | No | Yes |
| Review / Finalize | Validated candidates before master write | No direct overwrite | Uses saved revision |
| Item/Term master | Real PartCatalog/SAP-side records in `[SBOQTEC].[dbo].[@POITM]` / `[@PITM1]` | Controlled by final rules | No, separate master flow |

PartCatalog master write and SAP master write are the same action for Item/Term:
both target `SBOQTEC.dbo.@POITM` and `SBOQTEC.dbo.@PITM1`. AIX mirror writes are
sandbox validation only.

## 2. Source And Mode Dimensions

| Dimension | Values | Notes |
|---|---|---|
| `SourceType` | `MANUAL`, `AXON_AWARDED` | Manual is implemented first; AXON view later imports into same workspace |
| `CalculationMode` | `SINGLE`, `BULK` | Single has one line/no cross-line allocation; Bulk allocates header costs |

## 4. Workspace Location Labels

Use these labels when deciding where each Item/Term field belongs.

| Label | Meaning |
|---|---|
| `Main Grid` | Field should be visible/editable in the Workspace line grid because sales needs it during calculation or line review. |
| `Step 1 Default` | Run-level default that can be copied to lines, but line values remain the calculation truth. |
| `Review/Finalize` | Field is needed before real Item/Term master write, but it should not clutter the calculation grid. |
| `System` | Generated by backend/DB or inherited from saved revision metadata. |
| `Trace Only` | Stored for audit/transparency; not a direct Item/Term master field. |
| `Deferred` | Future scope or dependent on AXON/business confirmation. |

Important: the Workspace snapshot must preserve enough values to become
Item/Term candidates. The main grid should still stay focused on frequent
calculation and supplier-line work.

Calculation policy gaps:

1. Exchange rate date/source must be stored.
2. Document fee basis must be explicit: per each, item total, lot/batch service
   line, or other approved basis.
3. SCC / ASP must be confirmed as header-level, line-level, or both.
4. Zone/Freight rate lookup policy must be confirmed.
5. Weight-missing policy must be enforced before Bulk allocation.

- If any weight-based costs (PKH, SOC, Freight, CC) are greater than 0, every selected line must have a usable weight.
- If any selected line is missing weight/dimensions under this condition, CAL is blocked immediately to prevent cost distortion.
- Sales override must remain possible and must not be overwritten by AI/CWeight suggestions.

## 10. Current Open Decisions

These must be answered before schema rebuild is final:

| Question | Recommended default |
|---|---|
| Draft persist between page closes? | Yes, as overwriteable draft; revisions only on Save Revision |
| Separate Save Draft / Save Revision / Finalize? | Yes, keep the three actions distinct |
| Some Bulk lines missing weight | Allow CAL with warning only if approved; block Finalize |
| Every Bulk line missing weight | Block CAL |
| FR/CC/TT mixed currency from supplier | Defer mixed-currency engine; user normalizes first |
| Lot/batch document fee | Default to service line; override only with explicit warning |
| Existing item + existing term | Start with "add new term" before allowing update |
| Manual Finalize timing | Manual can move to Review/Finalize without AXON; master write still gated by final rules |
| Valid To default | Keep open until business confirms |
| SCC / ASP basis | Confirm whether run-level, line-level, or both |
